# 认证模块设计方案

## 1. 文档目标

本文档用于约定低代码平台认证模块的总体设计、数据模型、接口契约、安全策略、终端适配、刷新令牌治理和后续扩展方向。后续实现认证模块、租户接入和账号会话治理时，必须优先遵守本文档，除非安全要求、部署架构或业务边界发生变化并重新评审。

认证模块的目标如下：

- **身份可信**：所有受保护接口必须基于后端可验证身份访问，禁止依赖前端传入的账号、用户或租户字段。
- **终端统一**：PC 端和移动端使用同一套账号、租户、令牌和会话治理模型，差异只体现在客户端类型、令牌存储和安全策略上。
- **令牌可控**：采用 JWT 双令牌模式，同时对刷新令牌进行服务端持久化管理，支持查询、续期、废弃、强制下线和风险处置。
- **边界清晰**：认证模块只证明“是谁”和“当前属于哪个租户”，不实现角色、权限点、菜单权限、按钮权限或数据权限。
- **安全优先**：默认短生命周期访问令牌、刷新令牌轮换、服务端吊销、密码安全存储、审计日志和风险控制能力。
- **扩展可持续**：首版支持账号、租户关系和会话治理，后续可扩展短信登录、邮箱登录、单点登录、多因素认证和第三方身份源。

## 2. 总体结论

首版建议实现 **账号中心 + JWT 双令牌 + 服务端刷新令牌会话表 + 账号租户关系**。

推荐方案如下：

- 访问令牌 `accessToken` 使用 JWT，生命周期短，默认 15 分钟。
- 刷新令牌 `refreshToken` 使用高强度随机字符串或不透明令牌，服务端只存储哈希值，生命周期长，默认 7 天到 30 天。
- PC 端和移动端登录、刷新、退出接口保持统一，使用 `clientType` 区分终端。
- 登录成功后返回当前租户、账号信息、租户可访问列表和双令牌。
- 普通业务请求只携带访问令牌，由认证中间件验证签名、过期时间、租户范围和会话状态。
- 刷新访问令牌时必须校验刷新令牌服务端记录，刷新成功后轮换刷新令牌，旧刷新令牌立即废弃。
- 角色、权限点、菜单权限、按钮权限和数据权限后续由权限模块单独设计，认证模块只为权限模块提供可信账号和租户上下文。

不建议首版只使用纯无状态 JWT。原因是纯无状态 JWT 无法及时废弃刷新令牌、无法可靠强制下线、无法追踪终端会话，也无法满足移动端丢失设备、风控封禁、密码变更踢出等安全要求。

## 3. 术语定义

| 术语 | 说明 |
| --- | --- |
| 账号 | 可登录平台的身份主体，属于认证模块，不等同于 HRM 用户。 |
| HRM 用户 | 人力资源主数据中的人员信息，可与登录账号绑定，但不承担密码、登录和会话职责。 |
| 访问令牌 | 短期 JWT，用于访问业务接口。 |
| 刷新令牌 | 长期令牌，用于换取新的访问令牌和刷新令牌，必须服务端可管理、可废弃。 |
| 会话 | 一次终端登录产生的服务端刷新令牌记录，绑定账号、租户、终端、设备和状态。 |
| 认证 | 验证调用方身份是否可信。 |
| 账号租户关系 | 账号可访问租户的关系记录，用于登录选租户、切换租户和租户上下文校验。 |
| 认证上下文 | 认证中间件写入 `RequestContext` 的账号、租户、会话和终端信息。 |

## 4. 模块边界

### 4.1 首版包含能力

首版认证模块包含以下能力：

- 账号主数据管理。
- 密码登录。
- JWT 访问令牌签发和校验。
- 刷新令牌签发、轮换、查询、废弃和强制下线。
- PC 端和移动端统一登录、刷新、退出和当前用户接口。
- 账号与租户关系管理。
- 认证中间件写入 `RequestContext.user_id`、`RequestContext.auth_type`、`RequestContext.is_authenticated` 和租户来源。
- 安全审计日志，至少覆盖登录、退出、刷新、令牌废弃、密码修改和禁用账号。

### 4.2 首版暂不包含能力

首版暂不实现以下能力，但表结构和接口设计需要预留扩展：

- 短信验证码登录。
- 邮箱验证码登录。
- OAuth2 / OIDC 单点登录。
- LDAP、企业微信、钉钉、飞书等外部身份源。
- 多因素认证。
- Passkey 或 WebAuthn。
- 角色、权限点、账号角色授权。
- 菜单权限、按钮权限和数据权限。
- 细粒度字段权限。
- 动态策略引擎或 ABAC。
- 租户套餐、计费和用量限制。
- 跨租户超级管理员工作台的完整授权矩阵。

## 5. 账号与 HRM 用户关系

认证模块必须独立于 HRM 模块。

推荐关系如下：

```text
sys_accounts
  1 --- n sys_account_tenants
  1 --- 0..1 hrm_users
```

约束如下：

- `sys_accounts` 表达登录身份、密码凭证、账号状态和安全属性。
- `hrm_users` 表达人员主数据，例如工号、姓名、组织、岗位和任职关系。
- 一个账号可选绑定一个 HRM 用户，用于展示姓名、组织岗位、审批人、负责人等业务语义。
- 一个 HRM 用户是否允许登录，应由是否绑定启用账号决定，而不是 HRM 用户自身承担密码字段。
- 删除或禁用 HRM 用户时，不应直接物理删除账号；应通过认证模块执行禁用账号、废弃令牌和审计记录。
- 认证判断默认基于账号和租户，不直接基于 HRM 用户 ID。

### 5.1 手机号和邮箱边界

账号表和 HRM 用户表同时保存 `mobile`、`email` 容易产生语义歧义，例如同一人员换手机号后应更新哪一份、登录验证使用哪一份、通知发送以哪一份为准。首版必须避免这种双写问题。

推荐边界如下：

- HRM 用户表中的 `mobile`、`email` 表达人员主数据联系方式，用于组织通讯录、业务单据展示和人员资料维护。
- 认证账号首版不保存 `mobile`、`email`，只保存登录名、密码、安全状态和绑定 HRM 用户 ID。
- 登录名 `username` 可以由业务约定设置为工号、手机号、邮箱或自定义账号名，但认证模块不因此重复保存手机号或邮箱字段。
- 当前账号信息接口如需展示手机号或邮箱，应通过绑定的 HRM 用户摘要返回，字段语义明确为人员联系方式。
- 后续如果要支持短信登录、邮箱登录或安全通知，应新增账号联系方式表，而不是把 `mobile`、`email` 直接加回账号主表。

后续账号联系方式表建议命名为 `sys_account_contacts`：

| 字段 | 类型建议 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | `VARCHAR(64)` | 是 | 主键。 |
| `account_id` | `VARCHAR(64)` | 是 | 账号 ID。 |
| `contact_type` | `VARCHAR(32)` | 是 | 联系方式类型，例如 `mobile`、`email`。 |
| `contact_value` | `VARCHAR(128)` | 是 | 联系方式值，必须按类型规范化。 |
| `verified` | `BOOLEAN` | 是 | 是否已完成验证。 |
| `is_primary` | `BOOLEAN` | 是 | 是否主联系方式。 |
| `purpose` | `VARCHAR(32)` | 是 | 用途，例如 `login`、`notification`、`security`。 |
| 基础字段 | - | 是 | 统一基础字段。 |

该表属于认证模块后续扩展能力，首版不实现。

## 6. 令牌设计

### 6.1 双令牌模式

访问令牌和刷新令牌职责必须分离。

| 令牌 | 推荐形式 | 默认有效期 | 存储位置 | 用途 |
| --- | --- | --- | --- | --- |
| `accessToken` | JWT | 15 分钟 | PC 内存优先，移动端安全存储 | 访问业务接口。 |
| `refreshToken` | 随机不透明令牌 | 7 到 30 天 | PC HttpOnly Cookie 或安全存储，移动端 Keychain/Keystore | 换取新令牌。 |

访问令牌过期后，客户端调用刷新接口获取新的访问令牌。刷新令牌失效、废弃或过期时，客户端必须重新登录。

### 6.2 访问令牌 Claims

访问令牌 JWT 建议包含以下 claims：

| Claim | 说明 |
| --- | --- |
| `iss` | 签发方，例如 `taipus-api`。 |
| `sub` | 账号 ID。 |
| `tid` | 当前租户 ID。 |
| `sid` | 当前会话 ID，对应服务端刷新令牌记录。 |
| `aud` | 目标客户端或 API 受众，例如 `pc`、`mobile`、`api`。 |
| `iat` | 签发时间。 |
| `exp` | 过期时间。 |
| `jti` | 访问令牌唯一 ID，用于审计和极端场景黑名单。 |
| `auth_time` | 本次登录认证时间。 |
| `client_type` | 客户端类型，例如 `pc`、`mobile`。 |
| `scope` | 可选租户范围或认证能力摘要，首版不放权限列表。 |

不建议在 JWT 中放入以下信息：

- 密码、手机号、邮箱等敏感信息。
- 完整菜单树、完整权限点列表、角色列表或大体积组织范围。
- 可被前端篡改后影响授权的业务字段。
- HRM 用户详细资料。

### 6.3 签名算法和密钥

首版建议使用 `RS256` 或 `EdDSA` 非对称签名；如果部署复杂度需要降低，也可以先使用 `HS256`，但必须保证密钥高强度、定期轮换且只在后端可信环境保存。

建议：

- 生产环境优先使用非对称签名，私钥只在认证服务持有，业务服务只持有公钥。
- JWT header 必须固定允许算法白名单，禁止接受 `none` 或非预期算法。
- `kid` 用于密钥轮换，后端维护可用公钥集合。
- 密钥不得写入代码仓库、日志或前端配置。
- 密钥轮换时，新令牌使用新密钥，旧密钥保留到旧访问令牌自然过期后下线。

## 7. 刷新令牌治理

刷新令牌必须服务端持久化，支持管理和废弃。

### 7.1 刷新令牌状态

| 状态 | 说明 |
| --- | --- |
| `active` | 正常可用。 |
| `rotated` | 已轮换，旧令牌不可再用。 |
| `revoked` | 管理员或用户主动废弃。 |
| `expired` | 已过期。 |
| `compromised` | 疑似泄露，触发风险处置。 |

### 7.2 轮换规则

刷新接口必须执行刷新令牌轮换：

1. 校验刷新令牌格式和哈希。
2. 查询服务端会话记录。
3. 校验状态为 `active`。
4. 校验未过期、账号未禁用、租户可用、会话未强制下线。
5. 将旧刷新令牌状态更新为 `rotated`。
6. 创建新刷新令牌记录或更新同一会话的当前令牌哈希。
7. 签发新的访问令牌和刷新令牌。
8. 写入审计日志。

如果客户端使用已经 `rotated` 的旧刷新令牌再次刷新，应视为令牌重放风险：

- 将当前会话标记为 `compromised`。
- 废弃该账号同终端或全部终端刷新令牌，具体范围由安全策略决定。
- 返回 `-401`，要求重新登录。
- 记录安全审计日志。

### 7.3 废弃能力

刷新令牌管理必须支持以下废弃场景：

- 当前终端退出登录，只废弃当前会话。
- 用户修改密码后，废弃除当前会话外的其他会话。
- 管理员禁用账号后，废弃该账号全部会话。
- 租户禁用或暂停后，拒绝该租户下所有会话刷新和业务访问。
- 管理员在会话管理页面手动废弃指定终端。
- 风险控制识别异常后，废弃指定账号、租户、IP、设备或会话。

## 8. PC 端与移动端统一认证

### 8.1 统一接口

PC 端和移动端必须复用同一套认证接口，通过 `clientType` 区分终端。

| 场景 | 方法 | 路径 |
| --- | --- | --- |
| 登录 | `POST` | `/api/v1/auth/login` |
| 刷新令牌 | `POST` | `/api/v1/auth/refresh` |
| 退出当前终端 | `POST` | `/api/v1/auth/logout` |
| 退出全部终端 | `POST` | `/api/v1/auth/logout_all` |
| 当前账号信息 | `GET` | `/api/v1/auth/me` |
| 当前账号租户列表 | `GET` | `/api/v1/auth/tenants` |
| 切换当前租户 | `POST` | `/api/v1/auth/switch_tenant` |

接口路径必须使用 `snake_case`，JSON 字段必须使用 `camelCase`，响应统一使用 `ApiResponse<T>`。

### 8.2 客户端类型

`clientType` 首版建议支持：

| 类型 | 说明 |
| --- | --- |
| `pc` | Web PC 端。 |
| `mobile` | 移动端 H5、App 或小程序壳。 |
| `api` | 后续开放平台或系统集成预留。 |

服务端必须记录 `clientType`、`deviceId`、`deviceName`、`ip`、`userAgent` 和 `lastActiveTime`，便于用户查看登录设备和管理员排查风险。

### 8.3 PC 端存储建议

PC Web 推荐：

- `accessToken` 优先存储在内存，页面刷新后可通过刷新令牌恢复。
- `refreshToken` 优先使用 `HttpOnly + Secure + SameSite` Cookie。
- 如果因前后端部署限制必须由前端持有刷新令牌，必须降低刷新令牌有效期，并配合设备绑定、刷新轮换和异常重放检测。
- 不建议把长期刷新令牌存入 `localStorage`。

### 8.4 移动端存储建议

移动端推荐：

- 原生 App 使用系统安全存储，例如 iOS Keychain、Android Keystore。
- H5 或小程序场景按平台能力使用安全存储，避免明文长期保存。
- 设备 ID 必须由客户端稳定生成并传入，服务端只作为风险参考，不得单独作为身份凭证。
- 移动端刷新令牌可比 PC 端更长，但必须支持服务端废弃和重放检测。

## 9. 数据模型

所有持久化表必须包含统一基础字段：

```text
version
deleted
created_by
created_time
updated_by
updated_time
deleted_by
deleted_time
```

### 9.1 账号表 `sys_accounts`

| 字段 | 类型建议 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | `VARCHAR(64)` | 是 | 账号主键。 |
| `username` | `VARCHAR(64)` | 是 | 登录名，全局未删除数据唯一。 |
| `display_name` | `VARCHAR(128)` | 是 | 展示名称。 |
| `display_name_full_pinyin` | `VARCHAR(256)` | 是 | 展示名称全拼，用于搜索。 |
| `display_name_simple_pinyin` | `VARCHAR(128)` | 是 | 展示名称简拼，用于搜索。 |
| `password_hash` | `VARCHAR(255)` | 是 | 密码哈希，不存明文。 |
| `password_algo` | `VARCHAR(32)` | 是 | 密码哈希算法，例如 `argon2id`。 |
| `password_updated_time` | `TIMESTAMP` | 否 | 密码最后修改时间。 |
| `status` | `VARCHAR(32)` | 是 | 账号状态。 |
| `hrm_user_id` | `VARCHAR(64)` | 否 | 绑定 HRM 用户 ID。 |
| `last_login_time` | `TIMESTAMP` | 否 | 最近登录时间。 |
| `last_login_ip` | `VARCHAR(64)` | 否 | 最近登录 IP。 |
| 基础字段 | - | 是 | 统一基础字段。 |

账号状态建议：

| 状态 | 说明 |
| --- | --- |
| `enabled` | 正常可登录。 |
| `disabled` | 禁用，不允许登录和刷新。 |
| `locked` | 登录失败过多或风控锁定。 |
| `password_expired` | 密码已过期，登录后只能修改密码。 |

约束与索引：

- `username` 在未删除数据中全局唯一。
- 建立 `deleted + status` 索引，支持账号候选查询。
- 建立 `deleted + updated_time` 索引，支持分页查询。
- 建立展示名称和拼音字段查询索引。

### 9.2 账号租户关系表 `sys_account_tenants`

| 字段 | 类型建议 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | `VARCHAR(64)` | 是 | 主键。 |
| `account_id` | `VARCHAR(64)` | 是 | 账号 ID。 |
| `tenant_id` | `VARCHAR(64)` | 是 | 租户 ID。 |
| `status` | `VARCHAR(32)` | 是 | 关系状态。 |
| `is_default` | `BOOLEAN` | 是 | 是否默认租户。 |
| 基础字段 | - | 是 | 统一基础字段。 |

约束：

- `account_id + tenant_id` 在未删除数据中唯一。
- 同一账号最多一个有效默认租户。
- 只有 `enabled` 的关系可用于登录后选择租户。

### 9.3 刷新令牌表 `sys_refresh_tokens`

| 字段 | 类型建议 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | `VARCHAR(64)` | 是 | 会话 ID。 |
| `account_id` | `VARCHAR(64)` | 是 | 账号 ID。 |
| `tenant_id` | `VARCHAR(64)` | 是 | 当前会话租户 ID。 |
| `token_hash` | `VARCHAR(255)` | 是 | 刷新令牌哈希。 |
| `token_family` | `VARCHAR(64)` | 是 | 令牌族 ID，用于识别轮换链路。 |
| `status` | `VARCHAR(32)` | 是 | 刷新令牌状态。 |
| `client_type` | `VARCHAR(32)` | 是 | 客户端类型。 |
| `device_id` | `VARCHAR(128)` | 否 | 设备标识。 |
| `device_name` | `VARCHAR(128)` | 否 | 设备名称。 |
| `ip` | `VARCHAR(64)` | 否 | 最近使用 IP。 |
| `user_agent` | `VARCHAR(512)` | 否 | 最近使用 User-Agent。 |
| `expires_time` | `TIMESTAMP` | 是 | 过期时间。 |
| `last_used_time` | `TIMESTAMP` | 否 | 最近使用时间。 |
| `revoked_by` | `VARCHAR(64)` | 否 | 废弃操作人。 |
| `revoked_time` | `TIMESTAMP` | 否 | 废弃时间。 |
| `revoked_reason` | `VARCHAR(255)` | 否 | 废弃原因。 |
| 基础字段 | - | 是 | 统一基础字段。 |

安全要求：

- 数据库只保存刷新令牌哈希，不保存明文。
- `token_hash` 必须使用不可逆哈希，建议加入服务端 pepper。
- `token_hash` 在未删除数据中唯一。
- 建立 `account_id + status`、`tenant_id + status`、`expires_time` 和 `token_family` 索引。

### 9.4 审计日志表 `sys_auth_audit_logs`

| 字段 | 类型建议 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | `VARCHAR(64)` | 是 | 主键。 |
| `tenant_id` | `VARCHAR(64)` | 否 | 租户 ID，登录前可能为空。 |
| `account_id` | `VARCHAR(64)` | 否 | 账号 ID，登录失败时可能为空。 |
| `event_type` | `VARCHAR(64)` | 是 | 事件类型。 |
| `result` | `VARCHAR(32)` | 是 | `success` 或 `failure`。 |
| `client_type` | `VARCHAR(32)` | 否 | 客户端类型。 |
| `ip` | `VARCHAR(64)` | 否 | 客户端 IP。 |
| `user_agent` | `VARCHAR(512)` | 否 | User-Agent。 |
| `trace_id` | `VARCHAR(64)` | 是 | 请求链路 ID。 |
| `risk_level` | `VARCHAR(32)` | 否 | 风险等级。 |
| `message` | `VARCHAR(512)` | 否 | 安全事件摘要。 |
| 基础字段 | - | 是 | 统一基础字段。 |

审计日志默认不建议物理删除，应按合规和存储策略进行归档或分区。

## 10. 接口契约

### 10.1 登录

```text
POST /api/v1/auth/login
```

请求字段：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `username` | 是 | 登录名。 |
| `password` | 是 | 密码。 |
| `tenantId` | 否 | 指定登录租户；为空时使用账号默认租户或返回可选租户列表。 |
| `clientType` | 是 | `pc` 或 `mobile`。 |
| `deviceId` | 否 | 设备标识。 |
| `deviceName` | 否 | 设备名称。 |

响应字段：

| 字段 | 说明 |
| --- | --- |
| `accessToken` | 访问令牌。 |
| `refreshToken` | 刷新令牌。 |
| `expiresIn` | 访问令牌剩余秒数。 |
| `refreshExpiresIn` | 刷新令牌剩余秒数。 |
| `tokenType` | 固定为 `Bearer`。 |
| `tenantId` | 当前租户 ID。 |
| `account` | 当前账号摘要。 |
| `tenants` | 当前账号可访问租户列表。 |

如果账号有多个租户且未传 `tenantId`，建议返回业务错误 `-422` 和可选租户列表，要求客户端选择租户后重新登录或调用明确的选择租户接口。

### 10.2 刷新令牌

```text
POST /api/v1/auth/refresh
```

请求字段：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `refreshToken` | 是 | 当前刷新令牌；如果使用 HttpOnly Cookie，可由 Cookie 提供。 |
| `clientType` | 是 | 客户端类型。 |
| `deviceId` | 否 | 设备标识，用于风险校验。 |

刷新成功后必须返回新的 `accessToken` 和新的 `refreshToken`。旧刷新令牌立即不可再用。

### 10.3 退出

```text
POST /api/v1/auth/logout
```

退出当前终端必须废弃当前会话刷新令牌。访问令牌如果仍未过期，可以通过会话状态校验或短生命周期自然失效。对于高安全接口，可要求每次检查 `sid` 对应会话状态。

### 10.4 当前账号

```text
GET /api/v1/auth/me
```

返回当前账号、当前租户、可访问租户和绑定 HRM 用户摘要。该接口用于 PC 和移动端启动后恢复登录态。

### 10.5 切换租户

```text
POST /api/v1/auth/switch_tenant
```

切换租户必须重新签发访问令牌和刷新令牌，不能让客户端通过 `X-Tenant-Id` 任意覆盖 token 租户。

请求字段：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `tenantId` | 是 | 目标租户。 |
| `refreshToken` | 是 | 当前刷新令牌。 |

校验规则：

- 当前账号必须拥有目标租户有效关系。
- 目标租户必须启用。
- 原刷新令牌必须有效。
- 成功后废弃原会话或轮换为目标租户会话。

## 11. 认证中间件

认证中间件负责从请求中解析访问令牌、校验令牌并写入上下文。

推荐处理流程：

1. 跳过公开接口，例如健康检查、登录、刷新、公开资源。
2. 读取 `Authorization: Bearer <accessToken>`。
3. 校验 JWT 签名、算法、`iss`、`aud`、`exp`、`nbf` 和 `jti`。
4. 读取 `sub`、`tid`、`sid` 和 `client_type`。
5. 校验账号存在、未删除、状态可用。
6. 校验租户存在、未删除、状态可用。
7. 校验账号拥有该租户访问关系。
8. 按安全等级决定是否校验 `sid` 对应刷新令牌会话仍有效。
9. 写入 `RequestContext`。

`RequestContext` 应被写入：

| 字段 | 来源 |
| --- | --- |
| `user_id` | JWT `sub`。 |
| `tenant_id` | JWT `tid`。 |
| `tenant_source` | 固定为 `token`。 |
| `auth_type` | `User`。 |
| `is_authenticated` | `true`。 |

认证中间件接入后，租户中间件必须优先信任 token 中的租户。若请求同时携带 `X-Tenant-Id` 且与 token 租户不一致，必须返回 `-403`。

## 12. 与权限模块的边界

认证模块不实现授权模型。角色、权限点、菜单权限、按钮权限和数据权限必须由后续权限模块单独设计。

认证模块只向权限模块提供以下可信输入：

| 输入 | 说明 |
| --- | --- |
| `accountId` | 当前认证账号 ID，来自 JWT `sub`。 |
| `tenantId` | 当前认证租户 ID，来自 JWT `tid`。 |
| `sessionId` | 当前会话 ID，来自 JWT `sid`。 |
| `clientType` | 当前客户端类型，来自 JWT 或刷新令牌会话。 |
| `isAuthenticated` | 当前请求是否已完成认证。 |

权限模块不得信任前端传入的账号、角色、权限或租户字段，必须基于认证上下文和服务端授权数据判断访问权限。认证模块也不得把角色列表、权限点列表或菜单树写入 JWT，避免令牌过大、权限变更难以及时生效。

## 13. 租户集成

认证模块落地后，租户来源优先级调整为：

```text
JWT tid
> 可信内部调用上下文
> 受控 X-Tenant-Id
> 默认租户
```

规则：

- 普通登录用户的业务请求必须以 JWT `tid` 作为最终租户。
- body、query 中的 `tenantId` 不得覆盖上下文租户。
- `X-Tenant-Id` 只允许开发、内部调用或未登录公开接口使用。
- 切换租户必须通过认证接口重新签发令牌。
- 租户禁用或暂停时，访问令牌校验和刷新令牌校验都必须失败。

## 14. 安全策略

### 14.1 密码安全

- 密码必须使用 `argon2id` 或同级别密码哈希算法。
- 禁止保存明文密码、可逆加密密码或简单哈希密码。
- 密码输入错误必须使用统一错误提示，避免枚举账号是否存在。
- 登录失败次数达到阈值后应短暂锁定账号或触发验证码。
- 修改密码后必须废弃其他会话刷新令牌。

### 14.2 登录安全

- 登录接口必须限流，维度至少包含 IP、账号、租户和设备。
- 登录失败日志不得输出明文密码。
- 登录成功和失败都必须记录审计日志。
- 异常登录可触发风险标记，例如异地、设备变化、失败次数异常。
- 管理员重置密码后，账号下次登录应强制修改密码。

### 14.3 JWT 安全

- 访问令牌必须短有效期。
- 刷新令牌必须服务端可废弃。
- JWT 校验必须固定算法白名单。
- 必须校验 `iss`、`aud`、`exp` 和签名。
- 不得把敏感信息放入 JWT。
- 服务端日志不得输出完整令牌，最多输出 `jti`、`sid` 或脱敏片段。

### 14.4 CSRF 与 XSS

如果 PC 端刷新令牌使用 Cookie：

- Cookie 必须设置 `HttpOnly`、`Secure`、`SameSite=Lax` 或更严格策略。
- 跨站场景必须增加 CSRF token 或双提交校验。
- 访问令牌尽量放内存，减少 XSS 后长期令牌泄漏风险。

如果刷新令牌由前端显式保存：

- 必须缩短有效期。
- 必须启用刷新令牌轮换和重放检测。
- 必须限制 CSP，减少 XSS 风险。

### 14.5 租户与认证安全

- 跨租户访问必须返回 `-403` 或 `-404`，不得泄露其他租户资源是否存在。
- token 中的租户必须优先于普通 header，不允许前端通过 `X-Tenant-Id` 覆盖登录态租户。
- 账号禁用、租户禁用或会话废弃后，刷新令牌必须立即不可用。
- 访问令牌短期内可能仍有效，高安全接口可按 `sid` 查询服务端会话状态。

## 15. 错误码建议

认证模块复用现有业务错误码：

| 业务码 | 场景 |
| --- | --- |
| `-400` | 参数缺失、格式错误、令牌格式非法。 |
| `-401` | 未登录、访问令牌过期、刷新令牌无效、会话失效。 |
| `-403` | token 租户与请求租户冲突、禁止 header 覆盖租户。 |
| `-404` | 账号、账号租户关系、会话不存在或已删除。 |
| `-409` | 乐观锁版本冲突。 |
| `-422` | 账号禁用、租户暂停、状态流转非法、密码策略不满足。 |
| `-429` | 登录、刷新或敏感接口触发限流。 |
| `-500` | 系统错误、密钥加载失败、数据库不可用。 |

业务 API HTTP 状态码继续保持 `200`，健康检查和基础设施接口除外。

## 16. 缓存与性能

认证是高频路径，必须前置考虑性能。

建议：

- JWT 签名校验本地完成，不应每次访问业务接口都查数据库。
- 对普通业务接口，可缓存账号状态、租户状态和账号租户关系，缓存 TTL 建议 1 到 5 分钟。
- 对高风险接口，例如修改密码、切换租户、退出全部终端，可强制查询数据库确认会话状态。
- 账号禁用、租户禁用、账号租户关系禁用时必须主动失效相关缓存。
- 刷新令牌表按 `expires_time` 定期清理或归档过期数据。
- 审计日志可异步写入，但登录、刷新、废弃令牌等安全关键事件必须保证最终可追踪。

## 17. 实现建议

后端建议新增：

```text
backend/src/modules/auth/
  mod.rs
  model.rs
  dto.rs
  repository.rs
  service.rs
  handler.rs
  route.rs
  password.rs
  token.rs
```

职责：

- `model.rs`：平铺定义账号、账号租户、刷新令牌和审计日志模型。
- `dto.rs`：定义登录、刷新、退出、当前用户、当前账号租户和会话管理 DTO。
- `repository.rs`：使用 SQLx 显式 SQL，默认过滤 `deleted = false`。
- `service.rs`：承载登录、刷新、退出、切换租户、令牌废弃和事务编排。
- `password.rs`：封装密码哈希、密码校验和密码策略。
- `token.rs`：封装 JWT 签发、校验、密钥加载和刷新令牌生成。
- `handler.rs`：只负责参数提取、显式校验、调用 service 和返回 `ApiResponse<T>`。
- `route.rs`：挂载 `/api/v1/auth` 相关接口。

实现约束：

- 新增、修改、逻辑删除必须维护统一基础字段和 `version`。
- 所有名称字段必须生成拼音字段，后端不信任前端传入拼音。
- 所有写操作必须使用乐观锁或明确事务边界。
- 刷新令牌轮换必须放在数据库事务中完成。
- 密码哈希、令牌生成、密钥读取不得在日志中输出敏感值。
- MySQL 与 PostgreSQL migration 必须同版本维护。

## 18. 需要扩展的能力

建议按阶段扩展。

### 18.1 第二阶段

- 账号会话管理页面。
- 密码策略配置，例如长度、复杂度、过期周期和历史密码禁止复用。
- 登录失败锁定和图形验证码。
- 账号联系方式表，支持已验证手机号、邮箱、安全通知和后续验证码登录。

### 18.2 第三阶段

- 多因素认证。
- 短信、邮箱验证码登录。
- OAuth2 / OIDC 单点登录。
- LDAP 或企业身份源同步。
- 设备信任和异常登录提醒。
- 管理员模拟登录审计。

### 18.3 长期能力

- 租户套餐和功能开关。
- 零信任网关集成。
- 密钥管理系统和自动轮换。
- 安全报表、合规审计和 SIEM 对接。

## 19. 测试验收

### 19.1 Migration 测试

- MySQL/PostgreSQL 同版本认证 migration 存在。
- 所有认证表包含统一基础字段。
- 刷新令牌表只保存哈希字段，不保存明文。
- 账号、账号租户关系、刷新令牌和审计日志唯一约束符合文档。
- 逻辑删除后允许复用按未删除范围唯一的编码。

### 19.2 Service 测试

- 密码登录成功后签发双令牌。
- 密码错误返回统一错误，不泄露账号是否存在。
- 禁用、锁定、删除账号不能登录和刷新。
- 禁用、暂停租户不能登录、访问和刷新。
- 刷新令牌成功后旧令牌不可再用。
- 重放已轮换刷新令牌会标记风险并废弃会话。
- 退出当前终端只废弃当前会话。
- 退出全部终端废弃账号全部会话。
- 修改密码废弃其他会话。

### 19.3 API 测试

- 登录、刷新、退出、当前用户接口统一返回 `ApiResponse<T>`。
- JSON 字段使用 `camelCase`。
- 路径使用 `snake_case`。
- 缺少访问令牌返回 `-401`。
- 访问令牌过期返回 `-401`。
- token 租户与 header 租户冲突返回 `-403`。
- 业务错误 HTTP 状态码保持 `200`。

### 19.4 安全测试

- JWT `alg=none` 被拒绝。
- 非白名单算法被拒绝。
- 过期 JWT 被拒绝。
- 篡改签名 JWT 被拒绝。
- 刷新令牌明文不会进入数据库和日志。
- 登录接口限流生效。
- XSS 和 CSRF 风险在 PC 端令牌存储方案中有对应防护。

## 20. 默认假设

- 首版认证模块只实现账号密码登录，不实现短信、邮箱、SSO 和 MFA。
- 首版使用共享库共享表多租户方案，业务租户由 JWT `tid` 确定。
- 账号是登录主体，HRM 用户是人员主数据，二者通过可选绑定关系连接。
- 账号表首版不保存手机号和邮箱，人员联系方式以 HRM 用户为准。
- PC 端和移动端复用同一套接口、账号和会话模型。
- 刷新令牌必须服务端持久化并支持废弃，不采用纯无状态刷新 JWT。
- 权限体系后续由独立权限模块实现，认证模块只提供可信账号和租户上下文。
