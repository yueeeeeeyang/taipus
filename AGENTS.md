# AGENTS.md

## 项目目标

本项目目标是实现一个高性能、稳定、可扩展的低代码平台。所有设计与实现必须优先考虑长期维护、运行稳定性、性能容量、数据一致性和跨端体验一致性。

## 技术栈

- Web 后端必须使用 Rust、Axum、SQLx、Refinery。
- 数据库默认使用 MySQL，并保持对 PostgreSQL 的兼容能力。
- Web 前端必须使用 React、TypeScript、Ant Design、Ant Design Pro。
- 移动端必须使用 React、Ant Design Mobile。
- 如需引入新技术、框架或基础设施，必须先说明原因、收益、风险和替代方案，并在需求确认后再实施。

## 通用开发规则

- 所有代码必须添加完整、详细、有效的中文注释，注释应解释业务意图、关键约束、边界条件和非显而易见的实现原因。
- 必须为模块、公共类型、公共函数、复杂分支、错误处理、数据库迁移、权限边界、并发控制、性能约束和跨端接口契约添加中文注释。
- 简单语法行、闭合括号、直接字段赋值、显而易见的导入语句不强制逐行注释，避免为了注释覆盖率牺牲可读性。
- 禁止添加没有信息量的废话注释，例如仅重复变量名、函数名、字段名或语法行为的注释。
- 需求、边界、数据结构、权限模型、性能目标或交互行为不清晰时，必须先与需求方沟通确认，再开始实现。
- 优先复用项目内统一工具包、公共组件、公共类型、公共错误处理和公共基础设施。
- 优先选择成熟、活跃、稳定的第三方库，避免重复造轮子。
- 任何变更都必须尽量控制影响面，不得顺手重构无关代码。
- 所有实现必须考虑可测试性、可观测性、错误恢复和未来扩展。

## 后端开发规则

- 后端服务必须基于 Rust 与 Axum 构建，保持清晰的模块边界和分层结构。
- 数据访问必须优先使用 SQLx，避免绕过统一的数据访问约定。
- 数据库迁移必须统一使用 Refinery 管理，并在项目启动时自动运行。
- 接口输入必须进行显式校验，禁止依赖数据库错误作为业务校验。
- 接口输出必须保持稳定的数据结构，避免无计划破坏前端或移动端兼容性。
- 错误处理必须区分业务错误、参数错误、权限错误、资源不存在、并发冲突和系统错误。
- 涉及并发写入的数据必须考虑幂等性、事务边界、锁策略和版本冲突处理。
- 涉及批量查询、列表查询、搜索或统计的接口，必须前置考虑索引、分页、过滤条件和最大返回规模。

## 数据库与迁移规则

- 所有数据库表创建、修改、删除都必须通过 Refinery migration 完成。
- 所有默认数据、初始化数据和基础字典数据导入都必须通过 Refinery migration 完成。
- 禁止通过手工 SQL、临时脚本或应用启动外的隐式逻辑修改正式数据库结构。
- migration 必须可重复追踪、可审计，并能表达清晰的变更意图。
- 表结构设计必须同时考虑 MySQL 与 PostgreSQL 的类型兼容性、索引兼容性和命名兼容性。
- 字段命名应保持清晰稳定，避免使用数据库关键字或强绑定特定数据库方言的设计。
- 重要查询路径必须在建表或变更时同步规划索引。
- 删除业务数据时默认采用逻辑删除，不得直接物理删除，除非需求明确允许并说明风险。

## 前端开发规则

- Web 前端必须优先复用 Ant Design 与 Ant Design Pro 的组件、布局、表单、表格、导航和主题能力。
- 前端实现必须兼容暗黑模式，不得写死只适配亮色主题的颜色、阴影或背景。
- TypeScript 类型必须清晰表达接口结构、业务状态和组件入参，禁止滥用 `any`。
- 页面和组件必须优先复用公共布局、公共请求封装、公共错误处理和公共权限逻辑。
- 表单必须提供明确的校验规则、错误反馈和提交状态。
- 列表页必须前置考虑分页、筛选、排序、加载态、空状态和错误态。
- 涉及复杂交互时，应优先使用成熟组件和已有模式，避免自定义不稳定交互。

## 移动端开发规则

- 移动端必须使用 React 与 Ant Design Mobile。
- 移动端页面必须优先考虑触控体验、弱网体验、加载状态、空状态和错误恢复。
- 移动端交互不得直接照搬桌面端复杂表格或密集表单，应按移动端场景重新组织信息层级。
- 移动端请求、鉴权、错误处理和数据模型应尽量与 Web 前端保持一致，避免重复实现不兼容逻辑。

## 实体基础字段规范

所有实体、数据库表、后端模型和前端类型必须包含以下统一基础字段，除非该对象明确不是持久化业务实体：

- `version`：数据版本号，用于乐观锁、并发控制或变更检测。
- `deleted`：逻辑删除标记，用于区分有效数据与已删除数据。
- `createdBy`：创建人标识。
- `createdTime`：创建时间。
- `updatedBy`：最后更新人标识。
- `updatedTime`：最后更新时间。
- `deletedBy`：删除人标识。
- `deletedTime`：删除时间。

实现要求：

- 新增数据时必须正确写入创建字段和更新字段。
- 更新数据时必须正确维护更新字段和 `version`。
- 逻辑删除时必须正确维护 `deleted`、`deletedBy`、`deletedTime` 和 `version`。
- 查询默认必须过滤已逻辑删除的数据，除非接口语义明确要求包含已删除数据。
- 后端、数据库和前端类型应保持字段语义一致，避免同一字段在不同层含义不同。

## 接口命名规范

- 后端接口路径必须使用 `snake_case` 命名风格。
- 接口路径必须尽量短，避免冗长、重复和无意义层级。
- 路径中应使用资源名和必要动作名表达语义，避免把实现细节暴露到 URL。
- 批量、导入、导出、启停、排序等操作应使用清晰稳定的短路径命名。
- 一旦接口对前端或外部调用方开放，修改路径、字段或语义前必须评估兼容性影响。

## 性能与容量设计要求

- 性能与容量约束必须在需求分析和技术设计阶段前置考虑，不能等实现完成后再补救。
- 设计接口、表结构、索引、缓存、队列和批处理逻辑时，必须明确预期数据量、访问频率和增长趋势。
- 高频接口必须考虑查询次数、查询复杂度、连接池占用、序列化成本和响应体大小。
- 列表和搜索接口必须具备分页能力，并对最大分页大小设置合理上限。
- 批量写入、导入导出、异步任务和统计分析必须考虑超时、重试、幂等、进度反馈和失败恢复。
- 前端必须避免无意义重复请求、超大渲染列表和阻塞主线程的重计算。
- 移动端必须额外考虑网络波动、低性能设备和有限屏幕空间。

## 需求沟通规则

- 当需求存在歧义、缺少业务规则、缺少权限说明、缺少数据边界或缺少验收标准时，必须先提出具体问题。
- 沟通问题应聚焦会影响实现方案、数据结构、接口契约、权限边界、性能容量或用户体验的关键决策。
- 不得在核心业务语义不明确时自行假设并直接实现。
- 对于可合理推断且风险较低的细节，可以先说明默认假设，再继续推进。

## 交付前检查清单

交付任何代码或配置前，必须至少检查以下事项：

- 是否符合既定技术栈和项目结构约束。
- 是否存在完整、有效、不过度啰嗦的中文注释。
- 是否优先复用了统一工具包、公共组件和成熟第三方库。
- 持久化实体是否包含统一基础字段。
- 数据库结构和默认数据是否通过 Refinery migration 管理。
- 后端接口路径是否使用 `snake_case`，并保持足够短。
- 前端是否优先复用 Ant Design、Ant Design Pro 或 Ant Design Mobile。
- 前端是否兼容暗黑模式。
- 是否前置考虑性能、容量、索引、分页、缓存、并发和失败恢复。
- 需求不清晰处是否已经沟通确认或明确记录默认假设。
